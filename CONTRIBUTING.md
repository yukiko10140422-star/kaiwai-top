# KAIWAI 開発ガイド

このプロジェクトに参加してくれてありがとうございます。
ここでは、開発の流れやルールをまとめています。迷ったらこのページを見返してください。

---

## 開発の流れ（5ステップ）

### 1. Issue を選ぶ（または作る）

GitHub の **Issues タブ** を開いて、自分がやりたい Issue を選びます。
やりたいことがまだ Issue になっていなければ、新しく作ってOKです。

```
例: 「FAQセクションのデザインを改善したい」→ Issue を作成して自分をアサイン
```

> **ポイント**: 同じ Issue を複数人が同時に作業しないように、着手する前にコメントで「やります」と宣言しましょう。

### 2. ブランチを作る

**main ブランチから** 新しいブランチを作ります。main で直接作業しないでください。

```bash
# main を最新にする
git pull origin main

# 新しいブランチを作って移動する
git switch -c feature/faq-section
```

ブランチの名前は、下の **ブランチ命名規約** に従ってください。

> **Claude Code で作業している場合**: 「feature/faq-section というブランチを作って」と伝えれば、Claude が作ってくれます。

### 3. 作業する

コードを書いて、こまめにコミットします。

```bash
# 変更をステージに追加
git add src/components/FAQ.tsx

# コミットする
git commit -m "FAQセクションのデザインを改善"
```

> **Claude Code で作業している場合**: コードを書き終えたら「コミットして」と伝えるだけでOKです。

### 4. PR（Pull Request）を作る

作業が終わったら、GitHub に Push して PR を作ります。

```bash
# リモートに Push
git push -u origin feature/faq-section
```

Push したら GitHub 上で **Pull Request** を作成してください。
PR の書き方は、下の **PR のルール** を参照してください。

> **Claude Code で作業している場合**: 「PR を作って」と伝えれば、Claude が作成してくれます。

### 5. マージする

PR を作ると、自動でビルドチェックが走ります。
**ビルドが通ったら、自分でマージしてOK** です（セルフマージOK）。

ビルドが失敗した場合は、エラーを修正してから再度 Push してください。

> **Claude Code で作業している場合**: 「ビルドが失敗したので直して」と伝えれば、Claude が修正してくれます。

## ブランチ命名規約

ブランチ名は **種類/内容** の形式で作ります。内容は英語で、短くわかりやすく書きます。

| 種類 | 使うとき | 例 |
|------|---------|-----|
| `feature/〇〇` | 新しい機能を追加するとき | `feature/faq-section` |
| `fix/〇〇` | バグを修正するとき | `fix/hero-animation` |
| `improve/〇〇` | 既存の機能を改善するとき | `improve/cta-design` |

```
よい例:
  feature/faq-section     → FAQセクションを新規追加
  fix/hero-animation      → ヒーローのアニメーションバグを修正
  improve/cta-design      → CTAボタンのデザインを改善

悪い例:
  my-branch               → 何の作業かわからない
  test                    → 種類がない
  feature/あいうえお       → 日本語は使わない
```

## コミットメッセージ規約

コミットメッセージは **日本語** で書きます。
「何をしたか」がひと目でわかる一文にしてください。

### よい例

```
FAQセクションを追加
ヒーローのアニメーション崩れを修正
CTAボタンのデザインを改善
ナビゲーションのモバイル対応を実装
不要なコンソールログを削除
```

### 悪い例

```
修正           → 何を修正したかわからない
作業中          → 作業内容が不明
あ             → 意味がない
update         → 日本語で書いてください
```

## PR のルール

### タイトル

作業内容がわかる日本語で書いてください。

```
よい例: FAQセクションのデザインを改善
悪い例: 修正
```

### 本文

以下の2点を書いてください。

1. **何を変えたか** — 変更内容の説明
2. **確認してほしいポイント** — レビューアーに見てほしい箇所

```markdown
## 変更内容
- FAQセクションのアコーディオンUIを実装
- 開閉アニメーションを追加

## 確認ポイント
- モバイルでの表示崩れがないか
- アニメーションの速度が適切か
```

### セルフマージ

このプロジェクトでは **セルフマージOK** です。
ただし、以下の条件を満たしてからマージしてください。

- ビルドチェック（GitHub Actions）が通っている
- 明らかにおかしいところがない

## やってはいけないこと

### main ブランチで直接作業しない

必ずブランチを作ってから作業してください。
main に直接コミットすると、他の人の作業とぶつかる原因になります。

### 他の人のブランチを勝手に変更しない

他の人が作業中のブランチに、断りなくコミットしないでください。
変更したい場合は、まず相手に相談しましょう。

### .env ファイルをコミットしない

`.env` ファイルには秘密情報（APIキーなど）が含まれる場合があります。
**絶対にコミットしないでください。** `.gitignore` に登録されているので通常は問題ありませんが、注意してください。

## 困ったら

### Claude Code に聞く

全員が Claude Code で作業しているので、**わからないことは Claude に聞けばOK** です。

```
聞き方の例:
- 「コンフリクトが出たので解決して」
- 「ビルドが失敗したので原因を教えて」
- 「このエラーの意味を教えて」
- 「feature/faq-section というブランチを作って」
- 「変更をコミットして Push して」
- 「PR を作って」
```

Claude はこのプロジェクトのルールを把握しているので、ルールに沿った形でガイドしてくれます。

### Issue にコメントで相談する

技術的な相談や方針の確認は、GitHub の Issue にコメントで書きましょう。
他のメンバーにも状況が共有されるので、チーム全体で解決できます。

### それでもわからないとき

チームメンバーに直接聞いてください。わからないまま進めるより、聞いた方が早いです。
