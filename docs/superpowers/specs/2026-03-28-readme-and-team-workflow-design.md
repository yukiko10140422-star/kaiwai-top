# README & チーム開発ワークフロー設計

**日付:** 2026-03-28
**ステータス:** 承認済み

---

## 背景

KAIWAIプロジェクト（学生向けキャリア診断LP）を5人チームで開発するにあたり、以下を整備する：
- README.md（プロジェクト紹介）
- CONTRIBUTING.md（貢献ガイド）
- CLAUDE.md（Claude Code用ルール・メンバーガイド）
- Issue テンプレート・PRテンプレート
- ラベル体系・ブランチ保護ルール

## チーム特性

| 項目 | 内容 |
|------|------|
| 人数 | 5人 |
| 経験 | 全員初心者（Git未経験含む） |
| 作業環境 | 全員 Claude Code |
| 分担 | 固定なし、Issueを都度取る |
| Issue粒度 | 大きめ（機能単位） |
| マージ条件 | セルフマージ可（ビルド通過必須） |
| 言語 | 日本語（README・コミット・Issue すべて） |

## 採用アプローチ

**C案: フル構造化型** を採用。

理由：
- 5人チームで役割固定なしのため、ドキュメントで一貫性を担保する必要がある
- 全員Claude Code使用のため、CLAUDE.mdにルールを書けばClaudeがガイド役になる
- Issueテンプレート・PRテンプレートで初心者でも迷わず起票・PR作成できる

## 2. README.md

### 構成

1. **プロジェクト概要** — サービス説明（学生向けキャリア診断LP）、サイトURL
2. **技術スタック** — Next.js 16 / React 19 / TypeScript / Tailwind CSS v4、GitHub Pagesデプロイ
3. **セットアップ手順** — 必要なもの（Node.js 20以上, npm, Claude Code）、クローン→インストール→起動の3ステップ、コマンドはコピペ可能な形式
4. **プロジェクト構成** — ディレクトリツリー図、各コンポーネントの役割一覧表（13コンポーネント: Nav, Hero, Flow, Categories, GrowthPath, Features, Testimonials, FAQ, CTA, Footer, MobileFixedCTA, ScrollEffects, Divider）
5. **開発の進め方** — CONTRIBUTING.mdへのリンク、「Claude Codeを開いて作業開始、Claudeがガイドします」
6. **デプロイ** — main マージ → GitHub Actions → GitHub Pages の流れ図
7. **ライセンス** — 必要に応じて追記

### 設計方針
- 初心者が最初に見るファイルなので、コマンドはコピペできる形で記載
- コンポーネント一覧表でページ全体の構成が一目でわかるようにする
- 開発ルールの詳細はCONTRIBUTING.mdに分離し、READMEは短く保つ

## 3. CONTRIBUTING.md

### 構成

1. **開発の流れ（5ステップ）**
   1. Issueを選ぶ（または作る）
   2. ブランチを作る
   3. 作業する
   4. PRを作る
   5. マージする
   - 各ステップに具体例つき

2. **ブランチ命名規約**
   - `feature/〇〇` → 機能追加
   - `fix/〇〇` → バグ修正
   - `improve/〇〇` → 改善
   - 例: `feature/faq-section`, `fix/hero-animation`

3. **コミットメッセージ規約**
   - 日本語で書く
   - 「何をしたか」がわかる一文
   - 例: 「FAQセクションを追加」「ヒーローのアニメーション崩れを修正」

4. **PRのルール**
   - タイトル: 作業内容がわかる日本語
   - 本文: 何を変えたか・確認してほしいポイント
   - セルフマージOK（ビルド通過が条件）

5. **やってはいけないこと**
   - mainブランチで直接作業しない
   - 他の人のブランチを勝手に変更しない
   - .envファイルをコミットしない

6. **困ったら**
   - Claude Code に聞く（ルールはClaude が把握している）
   - コンフリクトが出たら「コンフリクトを解決して」とClaude に伝える
   - Issueにコメントで相談する

### 設計方針
- Git用語を最小限にし、動作ベース（「ブランチを作る」「PRを作る」）で説明
- 「やってはいけないこと」で初心者がハマるミスを先に防ぐ

## 4. CLAUDE.md（プロジェクト用）

### 構成

1. **プロジェクト概要** — サービス内容、技術スタック、デプロイ先

2. **チーム構成** — 5人、全員初心者、全員Claude Code使用

3. **Claudeの役割: メンバーのガイド**（核心部分）

   **作業開始時:**
   - 「何をしますか？」と聞き、対応するIssueがあるか確認
   - Issueがなければ作成を提案
   - mainブランチにいる場合、必ずブランチ作成を案内
   - ブランチ命名規約を案内（feature/, fix/, improve/）

   **作業中:**
   - コミットメッセージは日本語
   - .envファイルは絶対にコミットしない
   - 他の人のブランチは変更しない

   **作業完了時:**
   - PRの作成を案内する
   - ビルド確認（npm run build）を実行し、outディレクトリが正常に生成されることを確認してからPR作成
   - PRタイトル・本文は日本語
   - マージ方法を案内（セルフマージOK）

   **mainブランチ保護:**
   - mainで直接作業しようとしたら必ず止める

4. **技術ルール** — コンポーネント配置、テーマ変数、画像配置

### 設計方針
- Claudeへの指示形式で書き、メンバーが何も知らなくてもClaude が正しい手順に誘導
- 作業開始→作業中→作業完了の3フェーズで漏れなくカバー

## 5. Issueテンプレート

### feature.yml（機能追加）
- 入力項目: やりたいこと（必須）、詳細・参考情報（任意）
- 自動ラベル: `機能追加`

### bug.yml（バグ報告）
- 入力項目: 何が起きたか（必須）、期待する動作（必須）、スクリーンショット（任意）
- 自動ラベル: `バグ`

### improvement.yml（改善提案）
- 入力項目: どこを改善したいか（必須）、どう変えたいか（必須）
- 自動ラベル: `改善`

### 設計方針
- 入力項目は最小限（初心者が面倒に感じない）
- 自動ラベル付与で手動ラベル操作を不要に

## 6. PRテンプレート

```markdown
## 変更内容
（何を変えたか）

## 関連Issue
closes #

## 確認してほしいポイント
（特に見てほしいところ）

## スクリーンショット
（見た目が変わった場合は貼る）
```

### 設計方針
- `closes #` でマージ時にIssueが自動クローズされる
- スクリーンショット欄でUI変更の視覚的確認を促す

## 7. ラベル体系

| ラベル | 色 | 用途 |
|--------|-----|------|
| `機能追加` | `#0E8A16`（緑） | 新しい機能 |
| `バグ` | `#D73A4A`（赤） | 不具合修正 |
| `改善` | `#FBCA04`（黄） | 既存機能の改善 |
| `デザイン` | `#7057FF`（紫） | 見た目・UI関連 |
| `ドキュメント` | `#0075CA`（青） | README等の文書 |
| `対応中` | `#B0B0B0`（グレー） | 誰かが作業中 |

## 8. CIワークフロー（新規作成）

現在の `.github/workflows/pages.yml` はデプロイ専用（`on: push` のみ）のため、PR時のビルドチェック用に別途CIワークフロー `.github/workflows/ci.yml` を作成する。

```yaml
# ci.yml の概要
on: pull_request (branches: main)
steps: checkout → Node.js 20 setup → npm ci → npm run build
```

## 9. ブランチ保護ルール

`main` ブランチに対して:
- 直接push禁止
- PRを通してのみマージ可能
- レビュー承認は不要（セルフマージOK）
- CIワークフロー（ci.yml）のビルドチェック通過が必須

## 10. .gitignore の確認

`.env` が `.gitignore` に含まれていることを確認し、未記載の場合は追加する。初心者チームではうっかりコミットを仕組みで防ぐことが重要。

## 11. コンフリクト対処

5人が同じファイル（特に `page.tsx`）を編集する可能性があるため、CONTRIBUTING.md の「困ったら」セクションに以下を追記:
- コンフリクトが発生したらClaude Code に「コンフリクトを解決して」と伝える
- 自分の変更と相手の変更の両方を確認してからマージすること
